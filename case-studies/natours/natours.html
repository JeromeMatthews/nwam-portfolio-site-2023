PROJECT DETAILS

Satus: Complete

The Nature Tours (Natours) project was completed during 2021 - 2022 over a 5–6-month period where I was searching for a deeper understanding of how the back-end works in a modern web application development stack. I needed to understand by trying to build for myself a basic application that could carry out typical backend tasks while also providing a platform to learn and make use of APIs – (Application Programming Interfaces). JavaScript: Single thread processing and it’s limitations.

The single threaded operation of JavaScript and asynchronous coding concepts were a challenge for me to conceptualize for a significant portion of the first half of the course project. Understanding JavaScript’s use of a single thread in CPU processing wasn’t immediately obvious.
I came to get a better grapple on the concept once the theoretical section described the contrasts of how Node.js and the V8 engine works with respect to other pre-processors or languages that use parallel processing or multi-threaded processing, such as Java and PHP. Taking note of the point that JS was developed as a scripting language for the web and later adapted to work with operating systems as seen in node.js. And not essentially having the same demands that would be required of other languages initially intended to handle long running tasks like file uploads which can block other tasks from being carried out if multiple operations cannot be run concurrently in a program.
Once I started to understand that the asynchronous coding concept was an answer to the limitations the single threaded nature of JS had in trying to work with the operating system i.e.: the file system to prevent blocking processes that would slow down a program. It then started to stick a bit better.

Authentication and Authorization JWT and Building User Roles.

The difference between traditional server-maintained authentication and stateless client driven authentication.

APIs present a different model as it concerns security. In a more traditional development architecture, the responsibility of maintaining the “state” or in this case the data to confirm if a user should be able to access the server, is placed on the server. In a stateless system such as an API that responsibility remains on the user/client side in the form of the client’s local storage. One example is cookies on a web browser. This holds true for the idea that APIs can be RESTful - Representational State Transfers APIs. They provide the interfaces to work with their data, but aren’t concerned with maintaining any one state for the data provided.

Understanding how JWT works.

JsonWebTokens seemed a bit complex to get down, while implementation of the encoding wasn’t required in the course as it was already implemented in functions in the code. Understanding the underlying concepts were challenging. A token is made for the user, the user keeps this token throughout the use of the application (being logged in). Whenever the user wants to gain access to a page before their user permissions are checked (assigned to them on the creation of the user at sign up) their JWT needs to be authenticated.
In the token on the server, it’s unwrapped and examined to see if within it’s signature the required secret can be found. If the secret is found the data payload and its corresponding header are thought to be genuine and the user can be checked to see if they have the permissions to use a particular feature of the application.

Particularly working with new concepts like buffered and un buffered code, and interpolation within a templating engine.
